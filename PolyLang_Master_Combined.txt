Section 3 - Threat Model
Before writing a single line of security code, commit docs/THREAT_MODEL.md. Every security decision must trace back to a threat in this document.
Threat Actors
Malicious Script Author
Scripts designed to escape the sandbox, access the filesystem, crash the engine, consume unbounded resources, or exfiltrate data.
Malicious Plugin Consumer
A Godot project that loads PolyLang and abuses its API to run code outside the intended script execution path.
Malicious Vendored Dependency
Supply-chain tampering of pinned third-party code between pin time and use.
Compromised GitHub Actions Runner
Build environment tampering producing binaries that do not match the source.
Malicious Hot-Reload File
A file swapped on a watched path to inject code through the reload path.
Trust Boundaries
UNTRUSTEDAll script source code regardless of originUNTRUSTEDAll file paths received from the Godot editorUNTRUSTEDAll return values from foreign language runtimesUNTRUSTEDAll data crossing any FFI boundary in either directionPARTIALLY TRUSTEDGodot engine API calls (correctness only, not input validation)FULLY TRUSTEDOnly PolyLang core C++ code itself, after passing all 10 audit roundsEvery trust boundary crossing must be marked: // TRUST BOUNDARY: <what is validated and why>


Section 4 - Zero Trust Security Audit Protocol
After completing the full rebuild, run exactly 10 sequential zero-trust security audit rounds before delivering any code. Each round is independent. Do not skip rounds. Do not combine rounds. Each round must produce a written audit report before proceeding to the next.
Round 1 - FFI Boundary Audit
Finding prefix: VLN-R1-XXX
Inspect every FFI crossing. Check for: unchecked null returns, unvalidated string lengths, missing error codes, type mismatches across the ABI boundary, and any assumption that the foreign runtime is in a valid state.
NOTE: Fix ALL violations before proceeding to the next round.
Round 2 - Memory Safety Audit
Finding prefix: VLN-R2-XXX
Check for: use-after-free, double-free, heap buffer overflows, stack overflows in recursive dispatch, uninitialized memory reads, dangling pointers after hot-reload teardown, and any raw new/delete replaceable with RAII.
NOTE: Fix ALL violations before proceeding to the next round.
Round 3 - Hot-Reload Security Audit
Finding prefix: VLN-R3-XXX
Treat hot-reload as a completely untrusted code injection vector. Check for: missing file signature verification, no sandboxing of reloaded code, implicit trust of file paths, race conditions between teardown and reinit, leaked handles after failed reload, path traversal.
NOTE: Fix ALL violations before proceeding to the next round.
Round 4 - Input Validation Audit
Finding prefix: VLN-R4-XXX
Every string, path, identifier, and config value from outside must be validated before use. Check for: unvalidated script source strings, unvalidated file paths, missing UTF-8 validation, external input reaching FFI without sanitisation.
NOTE: Fix ALL violations before proceeding to the next round.
Round 5 - Vtable Integrity Audit
Finding prefix: VLN-R5-XXX
Check for: function pointers never validated before call, vtable slots overwritable after initialisation, missing null checks on all vtable entries, partial vtable initialisation treated as valid, vtable use after owning adapter is destroyed.
NOTE: Fix ALL violations before proceeding to the next round.
Round 6 - Concurrency & Thread Safety Audit
Finding prefix: VLN-R6-XXX
Check every shared resource for thread safety. Check for: unprotected global state, non-thread-safe adapters called from worker threads, missing locks on the language registry, hot-reload on a different thread from dispatch, non-atomic operations on shared counters.
NOTE: Fix ALL violations before proceeding to the next round.
Round 7 - Dependency & Supply Chain Audit
Finding prefix: VLN-R7-XXX
Check for: unpinned dependency versions, any dependency fetched over HTTP, any GitHub Action not pinned to a commit SHA, missing checksum verification on downloaded archives, dependencies with known CVEs.
NOTE: Fix ALL violations before proceeding to the next round.
Round 8 - Godot API Boundary Audit
Finding prefix: VLN-R8-XXX
Check for: missing return value checks on ClassDB registration, incorrect variant type assumptions, unvalidated object pointers from Godot, missing null checks on engine singletons, calls assuming a Godot object is alive without checking.
NOTE: Fix ALL violations before proceeding to the next round.
Round 9 - Error Handling & Fault Tolerance Audit
Finding prefix: VLN-R9-XXX
Check for: silent error swallowing, missing error propagation to GDScript, crashes instead of graceful degradation, adapters leaving the system in a corrupted state, assert used in production code paths.
NOTE: Fix ALL violations before proceeding to the next round.
Round 10 - Final Holistic Audit
Finding prefix: VLN-R10-XXX
Read the entire codebase as a fresh set of eyes. Check for: anything missed in rounds 1-9, architectural inconsistencies introduced during fixing, documentation no longer matching code, any TODO left in production code.
NOTE: Fix ALL violations before proceeding to the next round.


Section 5 - Architecture Requirements
Language Adapter System
• Every adapter must implement a C++20 concept called PolyLangAdapter enforcing the full interface at compile time. Missing methods = build failure with a clear error.
• Adapter interface must include: initialise(), shutdown(), execute(source, context), hot_reload(path), validate_source(source), get_language_id(), get_supported_extensions(), get_error_state(), reset_error_state().
• Every adapter must be completely self-contained in src/adapters/<language_name>/ with no cross-adapter dependencies.
• Adapters requiring a C shim must include the shim inside their own directory.
Language Registry
• Single source of truth: src/registry/language_registry.cpp
• Adding a new language requires editing exactly one file (the registry) and creating one new adapter directory. Nothing else.
• Registry must validate at startup that every adapter has a unique language ID. Duplicate IDs = hard startup failure.
String & Memory Management
• All string conversions must go through src/core/string_bridge.hpp. No adapter may perform its own string conversion.
• All memory crossing an FFI boundary must be owned and freed by the side that allocated it, documented in every FFI function signature.
• No raw new/delete anywhere. Use std::unique_ptr, std::shared_ptr, or arena allocators.
Hot Reload
• Hot reload must be a separate, isolated subsystem in src/core/hot_reload/.
• Every reload must validate the file path against an allowlist before touching the filesystem.
• Teardown and reinit must be atomic from Godot's perspective.
Error Handling
• Every function that can fail must return a PolyLangResult<T> type (std::expected-style wrapper).
• Errors must always propagate back to GDScript as a Dictionary with keys: success, error_code, error_message, adapter.
• No error may be silently swallowed anywhere.
Directory Structure
polylang/
├── .github/workflows/  (build.yml, test.yml, audit.yml)
├── src/
│   ├── core/           (adapter_concept.hpp, constants.hpp, error_types.hpp,
│   │                    polylang_result.hpp, string_bridge.hpp, logger.hpp)
│   ├── core/hot_reload/ (hot_reload_manager.hpp/cpp)
│   ├── registry/        (language_registry.hpp/cpp)
│   ├── adapters/<lang>/ (<lang>_adapter.hpp/cpp, shim/ if needed)
│   └── godot_entry.cpp  ← entry point ONLY, zero business logic
├── tests/
├── vendor/              (all vendored deps, pinned)
├── CMakeLists.txt
└── CMakePresets.json


Section 6 - Script Compilation Cache
• Cache keys: SHA-256 of script source + language ID + PolyLang version string.
• Location: user://polylang_cache/  |  TTL: polylang/cache/ttl_seconds (default 3600)
• Max size: polylang/cache/max_size_mb (default 256)  |  Eviction: LRU
• Disabled in debug builds by default. Cache corruption detected via checksums - silently discarded and recompiled, never a crash.


Section 7 - Debugging Support
• Every adapter implements get_stack_trace() returning Array of Dictionary with keys: file, line, function, language.
• Stack traces appear in Godot debugger panel formatted identically to GDScript stack traces.
• Every adapter implements get_current_line() for step debugging line highlighting.
• Breakpoint support: adapters declare POLYLANG_SUPPORTS_BREAKPOINTS. Supporting adapters implement set_breakpoint(file, line) and clear_breakpoint(file, line).
• Post-mortem dumps written to user://polylang_crashes/ on any fatal error.


Section 8 - Profiling & Performance Monitoring
• Every script execution wrapped in Godot profiler markers: PolyLang/<language>/<script_name>
• Profiler markers are zero-cost when profiler is not active.
• PolyLangProfiler singleton exposes: get_total_execution_time(language_id), get_call_count(language_id), get_slowest_script(), reset_stats()
• Per-frame budget monitoring: polylang/profiling/frame_budget_warning_percent (default 20)


Section 9 - Inter-Adapter Communication
• PolyLangBridge class registered with ClassDB exposing call_script(language_id, script_path, method_name, args).
• Cross-adapter calls go through the full validation pipeline - they are not trusted shortcuts.
• Type marshalling for cross-adapter calls goes through Godot's Variant system.
• Circular cross-adapter calls (A→B→A) detected and terminated with a clear error.
• Call depth capped: polylang/security/max_cross_adapter_depth (default 16)

Section 12 - Script Inheritance & Composition
• A PolyLang script must be attachable to any Godot Node or Resource as a script.
• Cross-language inheritance supported where adapter declares POLYLANG_SUPPORTS_GDSCRIPT_INHERITANCE.
• GDScript must be able to extend a PolyLang script.
• class_name equivalent supported in every adapter supporting static typing.


Section 13 - Async & Coroutine Support
• Every adapter declares POLYLANG_SUPPORTS_COROUTINES.
• Coroutines map to Godot's Signal await pattern.
• Coroutine execution respects the sandbox time limit - exceeded coroutines terminated cleanly.
• Orphaned coroutines whose owning node was freed must be detected and cleaned up.


Section 14 - Network Sandboxing
• polylang/security/allow_network_access defaults to false - hard default, cannot be overridden by script code.
• When false, any attempt by a foreign runtime to open a socket returns a permission denied error.
• When true, all network calls logged at INFO level via POLYLANG_AUDIT_LOG macro.
• HTTP/HTTPS is the only permitted protocol when network access is enabled.

Section 17 - Versioning

• MAJOR: adapter API breaks backward compatibility.  MINOR: new language adapter added.  PATCH: bug fixes, security patches, audit fixes.
• Every GitHub Actions release build tags the commit with v<MAJOR>.<MINOR>.<PATCH>.
• CHANGELOG.md maintained following keepachangelog.com conventions.


Section 18 - Logging & Diagnostics System
• Log levels: TRACE, DEBUG, INFO, WARN, ERROR, FATAL
• Debug builds: all levels active. Release builds: only WARN/ERROR/FATAL compiled in - others compiled out for zero runtime cost.
• Every log message includes: timestamp, log level, file name, line number, adapter name, message.
• Logs route to Godot's print/push_error system.
• POLYLANG_AUDIT_LOG macro always compiles in (even release) - writes to a separate audit log buffer accessible from GDScript.
• No std::cout or printf anywhere. All output goes through the logger.

Section 20 - Script Resource System
• Each language registers a custom ScriptLanguage implementation with Godot's ScriptServer.
• Scripts loadable via load("res://my_script.lua") or appropriate extension.
• Scripts support: attaching to nodes, exporting properties to inspector, emitting and connecting signals, calling into GDScript and back.
• Script compile/parse errors appear in the Godot editor error panel with file name and line number.


Section 21 - Property & Signal Exposure System
• Every adapter implements get_exported_properties() returning a typed list of PropertyInfo structs.
• Every adapter implements get_exported_signals() returning signal names and argument types.
• PropertyBridge utility in src/core/property_bridge.hpp provides helper functions for common type mappings.
• Mismatched property types produce a clear error at load time, not a crash at runtime.


Section 22 - Sandboxing & Execution Safety
• Maximum execution time per script call: polylang/security/max_execution_ms (default 5000ms). Scripts exceeding time limit terminated cleanly - never hang the engine.
• Scripts have no filesystem access unless adapter declares POLYLANG_ADAPTER_REQUIRES_FS_ACCESS and user enables it.
• Every adapter declaration file must have an explicit capabilities block listing exactly what system access it requires.


Section 23 - Editor Plugin

• Syntax highlighting for each registered language via Godot's CodeHighlighter.

• plugin.cfg version matches the main extension version exactly.


Section 24 - Extension Initialisation Order
The GDExtension entry point must follow this strict order. Deviation causes subtle runtime failures.
1. Initialise the logging subsystem (first - everything else depends on it)
2. Load and validate project settings
3. Initialise the string bridge
4. Initialise the type mapper
5. Initialise the compilation cache
6. Initialise the hot-reload manager (editor builds only)
7. Initialise the profiler (if enabled)
8. Validate all vendored dependency integrity checksums
9. Register all adapters with the language registry
10. Validate the adapter registry (check for duplicates, missing methods)
11. Initialise each adapter (failures logged + adapter disabled - do not abort whole init)
12. Register all Godot ClassDB classes
13. Register all project settings
14. Register the editor plugin (editor builds only)
15. Log successful initialisation with version string and active adapter list
NOTE: Shutdown must happen in exactly reverse order.


Section 25 - Crash Reporting & Recovery
• Every call into a foreign language runtime wrapped in a structured exception/signal handler.
• Linux/macOS: SIGSEGV and SIGABRT from a foreign runtime caught, adapter marked failed, control returned to Godot - not a full engine crash.
• Windows: Structured Exception Handling (SEH) wraps all foreign runtime calls.
• Post-mortem dumps written to user://polylang_crashes/ containing: timestamp, adapter name, script path, last known line, full stack trace, system info.
• error_recovered signal emitted on PolyLang singleton whenever a crash is caught and contained.


Section 26 - Backward Compatibility Policy
• GDScript-facing API stable from v1.0.0 onwards - no breaking changes without a MAJOR version bump.
• C++ adapter interface stable from v1.0.0 onwards - adding required methods requires a MAJOR version bump.
• Entry symbol polylang_library_init is permanently stable.
• Deprecations: deprecated in version N, removed in version N+2. Deprecated methods log a WARN on every call.

    // Phase 1: snapshot under SHARED lock (brief)
    void* old = nullptr; const PLAdapterVTable* vt = nullptr;
    { std::shared_lock lk(instance_lock_);
      old = foreign_instance_.load(std::memory_order_acquire); vt = vtable_; }
    // Lock RELEASED - foreign code can re-enter safely

    // Phase 3: swap under UNIQUE lock (brief)

      foreign_instance_.store(new_foreign, std::memory_order_release);
      if (old && vt && vt->pl_free_instance) vt->pl_free_instance(old); }
    // Phase 4: restore state OUTSIDE any lock
}
